---

Factory Flow supports single sign-on (SSO) through multiple identity providers. This allows your team to sign in using their existing corporate or social credentials.

## Supported providers

Factory Flow supports three SSO providers:

| Provider | Description | Best for |
|----------|-------------|----------|
| **Google** | Google Workspace and personal accounts | Teams using Google Workspace |
| **Microsoft** | Microsoft Entra ID (Azure AD) | Enterprise Microsoft environments |
| **GitHub** | GitHub accounts | Development teams |

## How SSO works

### User experience

1. User clicks "Continue with [Provider]" on the login page
2. User is redirected to the identity provider
3. User authenticates with their credentials
4. User is redirected back to Factory Flow, signed in

### Account linking

- First-time SSO users create a new Factory Flow account
- Email must be verified through the provider
- Users can link multiple providers to one account (if emails match)

## Google authentication

### For users

1. Click **Continue with Google** on the login page
2. Select your Google account
3. Authorize Factory Flow
4. You're signed in

### Supported account types

- Google Workspace accounts (corporate)
- Personal Gmail accounts

### Benefits

- No separate password to remember
- Inherits Google's security features
- Works with Google Workspace security policies

## Microsoft authentication

### For users

1. Click **Continue with Microsoft** on the login page
2. Enter your Microsoft credentials
3. Authorize Factory Flow
4. You're signed in

### Supported account types

- Microsoft Entra ID (Azure AD) accounts
- Microsoft 365 business accounts
- Personal Microsoft accounts

### Enterprise benefits

- Integrates with corporate identity management
- Supports conditional access policies
- Single identity across Microsoft ecosystem

### Configuration for IT admins

If your organization uses Microsoft Entra ID:

1. Users can sign in immediately with their work accounts
2. No additional configuration required in most cases
3. Admin consent may be required for first user from each tenant

## GitHub authentication

### For users

1. Click **Continue with GitHub** on the login page
2. Authorize Factory Flow
3. You're signed in

### Account requirements

- Active GitHub account
- Email must be verified on GitHub

### Use cases

- Development and engineering teams
- Technical users already using GitHub
- Organizations with GitHub Enterprise

## Setting up SSO for your organization

### Basic setup

SSO is enabled by default for all Factory Flow organizations. Users simply:

1. Visit the login page
2. Choose their preferred provider
3. Authenticate and join

### Inviting users with SSO

When inviting team members:

1. Send an invitation to their email
2. They click the invitation link
3. They can choose to sign in with any SSO provider
4. As long as the email matches, they join the organization

### Email matching

Factory Flow matches users by email address:
- User signs in with Google (john@company.com)
- Invitation was sent to john@company.com
- User is matched and joins the organization

## Security considerations

### Provider security

SSO inherits security from the identity provider:

| Feature | Managed by |
|---------|------------|
| Password policies | Identity provider |
| Multi-factor authentication | Identity provider |
| Session management | Identity provider |
| Account recovery | Identity provider |

### Factory Flow security

Factory Flow adds additional security:

- Email verification required
- Session timeouts
- Organization-level access control
- Role-based permissions

### Recommendations

1. **Enable MFA at the provider** - Adds an extra security layer
2. **Use corporate accounts** - Better audit and control
3. **Review access regularly** - Remove departed employees
4. **Limit personal accounts** - Where appropriate for your security policy

## Managing SSO users

### Viewing authentication methods

In **Organization Settings → Members**, you can see:
- Which users are in your organization
- Their roles and status
- When they last signed in

### Removing access

When a user leaves:

1. Remove them from the organization in Factory Flow
2. Optionally, disable their identity provider account
3. They lose access immediately

### Account recovery

If a user has issues signing in:

1. Verify their identity provider account is active
2. Check they're using the correct email
3. Have them try a different SSO provider if available
4. Contact support if issues persist

## Troubleshooting

### Common issues

| Issue | Solution |
|-------|----------|
| "Email already in use" | User has an existing account; use the same provider |
| Can't find organization | Check invitation email; request new invitation |
| SSO button doesn't work | Try a different browser; clear cache |
| Wrong account | Sign out of identity provider; try again |

### Provider-specific issues

**Google:**
- Check Google Workspace admin settings
- Verify third-party app access is allowed

**Microsoft:**
- Check Entra ID application permissions
- Verify admin consent for the application

**GitHub:**
- Ensure email is verified on GitHub
- Check organization SSO requirements if applicable

### Getting help

If you encounter SSO issues:
- Check your identity provider's status
- Verify your email is correct
- Try an incognito/private browser window
- Contact [support@factoryflow.com](mailto:support@factoryflow.com)

## Best practices

### For administrators

- Communicate supported SSO options to your team
- Recommend a preferred provider for consistency
- Document your authentication policy
- Regularly audit organization membership

### For users

- Use your corporate account when possible
- Enable MFA on your identity provider
- Report any suspicious activity
- Keep your email address current
